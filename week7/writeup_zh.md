## 任务 1：添加更多接口与校验
a. 相关提交 / issue 链接
>  https://app.graphite.com/github/pr/gabriellaXL/cs146s-assignments-gxl/1

b. PR 描述
> 本 PR 在第 7 周初始 API 的基础上，增强了输入校验、明确了错误处理，并为笔记（notes）和行动项（action items）补全了完整的 CRUD 接口。
>
> **问题：**
> 初始应用已有基本的列表 / 创建 / 更新功能，但 API 接口仍不完整，无法支撑健壮的 CRUD 工作流。此外，请求校验力度不足，路由层存在重复的查询/排序逻辑，部分框架用法已过时。测试 fixture 在 Windows 上也存在不稳定问题——临时 SQLite 文件有时在清理前未能正常释放。
>
> **解决思路：**
> 以收紧 API 契约为出发点，在此基础上逐步完善：
> - 对创建/更新请求体添加更严格的 Pydantic v2 校验，包括去除首尾空白和明确长度限制。
> - 将 `sort` 限制为显式白名单，而非对无效字段静默接受。
> - 补充缺失的接口：`GET /action-items/{id}`、`DELETE /notes/{id}`、`DELETE /action-items/{id}`。
> - 新增 `POST /notes/{id}/extract-action-items`，使提取逻辑可通过 API 接口调用，而不仅依赖内部辅助函数。
> - 抽取共用的 `get_or_404` 和 `apply_sort` 辅助函数，减少各路由间的代码重复。
> - 将 FastAPI 应用的启动路径从已废弃的 `on_event` 迁移至 `lifespan`。
> - 将时间戳默认值更新为带时区的 UTC 值。
> - 修复测试 fixture，改用 `TemporaryDirectory` 并在 teardown 时正确销毁 SQLAlchemy engine。
>
> **测试：**
> - 运行命令：`PYTHONPATH=. poetry run pytest -q backend/tests`
> - 结果：`5 passed`
> - 扩展的测试覆盖：
>   - 创建/获取/列表/修改/删除的正常流程
>   - 提取接口的行为
>   - 空值或非法请求体的校验失败场景
>   - 无效的分页和排序参数
>   - 访问不存在资源时的 `404` 行为
>
> **值得注意的权衡 / 后续工作：**
> - 使用白名单式排序字段，让行为更可预测、更易测试。这比静默回退到默认排序更严格，但能提供更好的 API 契约。
> - 提取接口目前只返回提取的字符串，不做持久化。这样可以让任务 1 聚焦于 API 结构和校验；更丰富的提取行为留待任务 2 演进。

## 任务 2：扩展提取逻辑
a. 相关提交 / issue 链接

b. PR 描述
> 本 PR 将笔记到行动项的提取流程，从演示性启发式规则升级为基于规则的结构化分析管道。
>
> **问题：**
> 原有提取逻辑仅能识别少数固定模式（`TODO:`、`ACTION:` 以及以 `!` 结尾的行）。它无法说明为什么某条内容被提取出来，无法附带元数据（如来源行号或截止日期提示），也过于脆弱，难以支撑真实的记笔记场景。
>
> **解决思路：**
> 用结构化规则引擎取代原有的临时提取器，保持实现的确定性和可测试性：
> - 新增专用的 `analyze_action_items()` 管道，返回结构化提取结果，而非仅返回原始字符串。
> - 保留简单的 `extract_action_items()` 包装函数，供只需规范化文本的调用方使用。
> - 扩展支持的模式，包括：
>   - 标签式行动行，如 `TODO:`、`ACTION:`、`FOLLOW-UP:`、`NEXT STEP:`
>   - 未完成的清单项，如 `[ ] prepare demo`
>   - 请求/承诺表达，如 `Please send the recap by Friday`
>   - 祈使句式任务，如 `Review the API contract`
> - 增加规范化和去重处理，避免重复的行动行产生重复提取结果。
> - 为每个提取项附加轻量元数据：
>   - `source_line`（来源行）
>   - `trigger`（触发模式）
>   - `confidence`（置信度）
>   - 可选 `due_hint`（截止日期提示）
>   - 可选 `assignee_hint`（负责人提示）
> - 更新提取 API，返回结构化提取记录，而非不透明的字符串。
>
> **测试：**
> - 运行命令：`PYTHONPATH=. poetry run pytest -q backend/tests`
> - 结果：`7 passed`
> - 新增测试覆盖：
>   - 单条笔记中的多种提取模式
>   - 触发模式、截止日期、负责人的元数据生成
>   - 重复任务的去重
>   - 已完成或不可行动行的排除
>   - 提取 API 返回结构化响应的行为
>
> **值得注意的权衡 / 后续工作：**
> - 提取器有意保持基于规则，而非使用 LLM，以确保行为可预测、成本低廉、易于回归测试。
> - 当前分析只生成轻量提示，而非完整的任务解析。如有需要，后续可在不改变核心提取架构的前提下，扩展更丰富的字段（如优先级或负责人规范化）。

## 任务 3：尝试添加新模型与关联关系
a. 相关提交 / issue 链接
> 待补充。在 PR 创建后，添加任务 3 的分支、提交记录和 PR 链接。

b. PR 描述
> 本 PR 引入新的 `Project` 模型，并通过一对多关联将其与 `ActionItem` 连接。
>
> **问题：**
> 改动前，行动项均为独立记录。应用没有办法将相关工作分组、按项目筛选任务，或在 API 和界面中体现模型关联关系。现有的 SQLite 设置也没有为已有数据库新增外键列提供迁移路径。
>
> **解决思路：**
> 添加 `Project -> ActionItem` 关联，并在全栈范围内更新应用：
> - 新增 `Project` 模型，包含 `name`、`description`、时间戳，以及与 `ActionItem` 的反向关联。
> - 在行动项的 schema 和接口中增加 `project_id` 和 `project` 支持。
> - 新增 `/projects` 路由，支持：
>   - 列表查询
>   - 创建
>   - 获取详情（含嵌套行动项）
>   - 修改
>   - 删除
> - 增强行动项的功能：
>   - 创建时可指定关联项目
>   - PATCH 时可重新指定或清除项目
>   - 支持按 `project_id` 筛选行动项
>   - 行动项响应中附带项目元数据
> - 为 SQLite 添加轻量 schema 升级路径，使旧数据库无需完整迁移框架即可接收新的 `project_id` 列和索引。
> - 更新种子数据和静态前端，使项目可在应用界面中创建和关联。
>
> **测试：**
> - 运行命令：`PYTHONPATH=. poetry run pytest -q backend/tests`
> - 结果：`9 passed`
> - 新增和更新的测试覆盖：
>   - 项目的 CRUD 行为
>   - 嵌套关联的读取
>   - 行动项与项目的关联和取消关联
>   - 项目名称唯一性校验
>   - 按 `project_id` 筛选行动项
>   - 删除项目后子行动项关联被清除
>
> **值得注意的权衡 / 后续工作：**
> - 选择单个 `Project -> ActionItem` 关联，而非更复杂的多对多设计，以保持模型简单易懂、易于测试。
> - 当前数据库演进路径有意保持轻量，适用于作业规模的应用。生产系统应使用 Alembic 或等效工具进行真正的数据库迁移。

> 本 PR 加强了应用中分页和排序行为的测试覆盖。
>
> **问题：**
> 早期测试只验证了列表接口能返回成功响应，但没有对排序顺序、页面边界，或过滤器与分页的组合行为进行有力断言。这导致多个重要的回归风险未被发现，例如错误的排序顺序、偏移量计算失误、空页处理，或过滤器应用顺序错误。

>   - 升序和降序排序
